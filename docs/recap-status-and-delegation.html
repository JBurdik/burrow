Recap — status dots & sub-agent delegation

What changed on 2026-06-03: status-indicator bug fixes + a reworked fire-and-forget delegation model. See also burrow.html and context.html.
TL;DR. Fixed the agent status dots (stuck-orange, false-blue, mid-conversation renames, and the real showstopper: a stale hook-server port after an app restart). Reworked sub-agent delegation from block-and-wait to fire-and-forget + burrow collect, added burrow sessions, and a soft per-workspace agent cap configurable in Settings.

Part 1 — Status indicator fixes

Bug A — stuck orange (running) after the agent finished

Cause

running was only cleared by a Stop hook or the shell returning to foreground. A lost best-effort POST or an interrupted turn left the spinner on forever.

Fix

Idle Notification now maps to done (clears it); the poll emits busy=false when the shell returns. (See also the port fix below — the dominant real cause.)

Bug B — done shows blue (waiting for user)

Cause

Claude Code fires Notification for two things: a permission prompt and ~60 s after a turn ends, idle at the prompt ("…waiting for your input"). The CLI mapped every Notification → waiting (blue), so a finished agent flipped blue. Worse, that trailing waiting clobbered the fresh done because the listener emitted busy+needsInput as two order-dependent signals.

Fix

burrow hook now disambiguates Notification by its message (idle ping → done, else → waiting). XTerm.vue forwards a single semantic agentState event, so ordering can't corrupt the state.

Bug C — terminal renamed mid-conversation

Cause

The 2 s poll treated an empty get_pty_foreground reading (a daemon race) as "idle" and reset the tab to "Terminal N"; child processes the agent spawned also stole the title.

Fix

Empty readings are ignored; the poll is now sticky per agent session (stays "Claude" across child processes until the shell genuinely returns).

Root cause #0 — stale hook-server port (the real showstopper)

Cause

The hook server binds a random port each launch (127.0.0.1:0). create_pty bakes that port into the PTY env (BURROW_HOOK_PORT). After an app restart, daemon-reattached PTYs keep the old port while the server moved to a new one — and burrow status trusted the env port first. Result: every reattached session POSTed to a dead port → the dot never updated at all, and any done sent after a restart was lost (→ permanent orange).

Fix

The live <app-data>/hook.port file is now authoritative (rewritten every launch); the env port is only a fallback. Diagnosed against the live app: server returned HTTP 200 but the installed CLI was POSTing to the stale port.

Status flow (after fixes)

agent hook fires
  └─ burrow hook  (maps hook_event_name → running|waiting|done; Notification by message)
       └─ burrow status <state>   (POST to hook.port — FILE first, env fallback)
            └─ hook HTTP server emits  pty-hook-{id}
                 └─ XTerm.vue → ONE agentState event
                      └─ Terminal.vue onAgentState → status dot
                         (running=orange spinner · waiting=blue · done=lime · review=green pulse)

Part 2 — Sub-agent delegation rework

Old model: burrow spawn --token t then burrow wait t — which blocked the spawning agent. New model: fire-and-forget + collect. Spawn agents, keep working, pull results when you want.

New / changed commands

CommandWhat it does
burrow spawnFire-and-forget (unchanged), now prints the soft per-workspace limit on success
burrow collect [tok…]New. Non-blocking: print + consume the results of finished tokens; report the rest as pending. No tokens = every finished sub-agent
burrow sessions [--count]New. List live daemon PTY sessions (pty id + cwd) via the daemon socket protocol
burrow waitKept as legacy blocking variant — prefer collect
burrow captureNow also calls burrow status done (per-launch --settings Stop hook overrode the global one → fixed stuck-orange sub-agents)

The recap pattern

burrow spawn --token a claude "audit src/auth for bugs"
burrow spawn --token b claude "audit src/api for bugs"
# …keep doing your own work — never block…
burrow collect a b      # prints whichever FINISHED, consumes them
burrow collect a b      # later: picks up the stragglers

Soft concurrency limit

Configurable in Settings → Agents → Max concurrent sub-agents (default 3, per workspace). The frontend publishes it via the new set_max_agents Tauri command to <app-data>/max_agents, which burrow spawn reads and surfaces. It is advisory — the /burrow skill is instructed to respect it; nothing hard-blocks a spawn.
