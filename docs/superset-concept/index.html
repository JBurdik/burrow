How Superset Detects Terminal/Agent Status

Concept study — reverse-engineered from superset-sh/superset, for comparison with Burrow's own status detection.

source: ~/.opensrc/repos/github.com/superset-sh/superset/main  ·  apps/desktop (Electron)

TL;DR. Superset does not parse PTY output, watch quiescence timers, or sniff the foreground process to decide if an agent is busy. It is 100% hook-driven: the running CLI agent curls a tiny local HTTP server on every lifecycle event, and the UI flips a 4-state status off those callbacks. Burrow, by contrast, uses an OSC 9998 escape channel + foreground-process polling fallback. Two opposite design choices.

The four states

apps/desktop/src/shared/tabs-types.ts:26–40

export type PaneStatus = "idle" | "working" | "permission" | "review";
export type ActivePaneStatus = Exclude<PaneStatus, "idle">;

export const STATUS_PRIORITY = {
  idle: 0,
  review: 1,
  working: 2,
  permission: 3,
} as const satisfies Record<PaneStatus, number>;
StateMeaningUI indicator
workingAgent processing / thinkingAmber, pulsing — "Agent working"
permissionAgent blocked, needs user input/approvalRed, pulsing — "Needs input"
reviewAgent done while user wasn't watchingGreen, static — "Ready for review"
idleNothing pending / user already engagedno indicator

The STATUS_PRIORITY map matters because a tab aggregates many panes — when a tab shows one badge for N panes, the highest-priority status wins (permission > working > review > idle).

Detection mechanism: HTTP lifecycle hooks

A local Express server runs inside the Electron main process. Each terminal is launched with env vars pointing at it.

apps/desktop/src/main/lib/terminal/env.ts:472–482

SUPERSET_PANE_ID:      paneId,
SUPERSET_PORT:         String(env.DESKTOP_NOTIFICATIONS_PORT),
SUPERSET_HOOK_VERSION: HOOK_PROTOCOL_VERSION,   // "2"

The agent's hooks (Claude Code hooks, Codex notify, etc.) fire a plain GET — no tty needed, works from any subprocess:

GET /hook/complete?eventType=Start&paneId=$SUPERSET_PANE_ID
GET /hook/complete?eventType=Stop&paneId=$SUPERSET_PANE_ID
GET /hook/complete?eventType=PermissionRequest&paneId=$SUPERSET_PANE_ID

Many agents, three canonical events

Different agents emit different hook names. mapEventType collapses dozens of aliases into Start / Stop / PermissionRequest, so one state machine handles Claude Code, Codex, and others.

apps/desktop/src/main/lib/notifications/map-event-type.ts

CanonicalAliases mapped in
StartStart, SessionStart, UserPromptSubmit, PostToolUse, PostToolUseFailure, BeforeAgent, AfterTool, task_started, snake/camel variants…

StopStop, agent-turn-complete, AfterAgent, session_end, task_complete
Clever bit: PostToolUse maps to Start, not Stop. So every tool call re-asserts "working" — the amber state stays alive across a long multi-tool turn instead of flickering to idle between steps.

The state machine

apps/desktop/src/renderer/stores/tabs/useAgentHookListener.ts:74–112

if (eventType === "Start") {
  state.setPaneStatus(paneId, "working");
} else if (eventType === "PermissionRequest" || eventType === "PendingQuestion") {
  state.setPaneStatus(paneId, "permission");
} else if (eventType === "Stop") {
  const isInActiveTab = isTabActive &&
    (getCurrentWorkspaceId() === workspaceId || isPaneFocused);

  const nextStatus =
    pane?.status === "permission" ? "idle"   // already engaged → clear
    : isInActiveTab              ? "idle"    // user is watching → no badge
    :                              "review"; // user away → green "look here"

  state.setPaneStatus(paneId, nextStatus);
}
# the lifecycle launch terminal → idle agent Start / tool → working (amber, pulsing) agent asks approval → permission (red, pulsing — highest priority) agent Stop, watching → idle agent Stop, away → review (green — "ready for review") click tab / focus → idle

"Agent thinking" vs "agent waiting for me" is not inferred — it's two distinct hook events (Start vs PermissionRequest). No heuristics, no false positives.

The gap: no Stop hook on Ctrl+C

The weakness of a pure-hook model: if the agent never fires Stop (user hits Ctrl+C, process killed, permission denied), the pane would be stuck on working forever. The fallback is the PTY-exit event, not output parsing.

apps/desktop/src/renderer/stores/tabs/useAgentHookListener.ts:114–123

} else if (event.type === NOTIFICATION_EVENTS.TERMINAL_EXIT) {
  // Clear transient status for unmounted panes
  if (currentPane?.status === "working" || currentPane?.status === "permission") {
    state.setPaneStatus(paneId, "idle");
  }
}

So a stuck "working" only clears when the whole terminal process exits — if the agent is interrupted but the shell stays alive, the amber badge lingers until the next event. Acknowledged limitation in the code comments.

What it deliberately does NOT do

  • No PTY output parsing for status. The headless emulator (headless-emulator.ts) tracks DECSET/DECRST modes, OSC-7 cwd, and screen snapshots — but only for restore fidelity on reattach, never for activity.
  • No quiescence timers ("no output for 2s ⇒ idle").
  • No foreground-process-name sniffing.
  • No OSC status escape codes on the PTY channel.

Superset vs Burrow

SupersetBurrow
Primary signalHTTP GET to local Express serverOSC 9998;running|waiting|done on PTY → /dev/tty
FallbackPTY exit event clears stuck stateget_pty_foreground polling every 2s (proc name + recent output)
Statesidle / working / permission / reviewbusy / waiting / done
"Done while away"distinct review state (green)—
Multi-agent normalizationmapEventType alias tableClaude-specific hooks JSON
Transport from no-tty subprocessworks natively (HTTP)OSC can't reach tty → file-based burrow spawn instead
Takeaway for Burrow. Superset sidesteps the exact problem Burrow hit — subprocesses with no controlling tty can't write OSC to /dev/tty. An HTTP loopback hook (or Unix socket) is tty-independent by construction, the same reason Burrow moved spawn/wait to a file channel. Two states Burrow lacks are worth stealing: permission (distinguish "waiting for approval" from generic idle) and review (agent finished while you were on another tab — the highest-value notification in a multi-agent UI).
