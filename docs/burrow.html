The burrow CLI

Let an AI agent running inside Burrow delegate work to sub-agents in new terminal tabs — and optionally wait for their results.

The idea

Every agent in Burrow runs inside a terminal tab. To delegate, burrow spawn drops a small request file in a directory Burrow watches; the app polls it and opens a new tab. Results come back through result files. No port, no socket, no MCP server.

Why files, not the terminal? An earlier design printed an escape sequence to /dev/tty. But agents invoke burrow through their Bash tool / hooks, whose subprocesses have no controlling terminal — so /dev/tty fails with "Device not configured" and Claude also captures the command's stdout, so the bytes never reach the PTY. File-based transport sidesteps both problems and works from any subprocess.
Subscription-safe by design. Sub-agents are launched interactively (plain claude, with the prompt as a positional arg) — never claude -p / --print and never the Agent SDK. This matches the post-2025-06-15 policy for subscription use. The CLI passes commands through verbatim and must never inject -p.

Commands

burrow spawn [--token T] [--cwd DIR] <command...>

Open a new terminal tab in the current project and run command in it. Fire-and-forget — returns immediately.

  • --token T — attach an id so you can burrow collect T for the result later.
  • --cwd DIR — run the new tab in DIR instead of the current project dir.
burrow spawn claude "write unit tests for src/foo"
burrow spawn codex  "refactor the auth module"

On success it prints the soft per-workspace concurrency limit (default 3, configurable in Settings → Agents, published to <app-data>/max_agents). The limit is advisory — the /burrow skill is told to respect it; spawn does not enforce it.

burrow collect [token...]

Non-blocking recap. For each given token that has finished (its <token>.done exists), print its final message and consume it (delete .done + .result), so a later collect only returns newly-finished ones. Tokens still running are reported as pending. With no tokens, collects every finished sub-agent in the session. This is the preferred pattern: fan out, keep working, then collect — never sit blocked. 

burrow spawn --token a claude "audit src/auth for bugs"
burrow spawn --token b claude "audit src/api for bugs"
# …do your own work…
burrow collect a b   # prints whichever FINISHED, consumes them; reports the rest as pending
burrow collect       # or: every finished sub-agent, no token list

burrow sessions [--count]

List the live PTY sessions the daemon is holding (pty id + cwd), or just their count with --count. Speaks the daemon's newline-JSON socket protocol (Auth then ListSessions) via python3, reading daemon.sock + daemon.token from BURROW_HOME_DIR.

burrow list-workspaces  /  burrow list-tabs [--ws ID] (read)

Inspect the app from the terminal. list-workspaces prints every workspace as <id>\t<name>\t<path>; list-tabs prints a workspace's tabs as <pty_id>\t<title> (default: the workspace this tab belongs to, or --ws ID for another). Both are read commands: they drop a request dir carrying a token, then block briefly polling <session>/<token>.result — exactly the channel burrow wait uses. They're answered entirely in Rust inside take_spawn_requests (a SQLite query writes the result file); the request never reaches the frontend. Parse the ids out of the output to feed the focus / new-tab commands. 

burrow list-workspaces           # 1\tagentic-ide\t/Users/me/code/agentic-ide
burrow list-tabs                  # 42\tclaude — fix auth bug
burrow list-tabs --ws 3

burrow focus-workspace <ID>  /  burrow focus-tab <ID>  /  burrow new-tab [--ws ID] [--cmd CMD]

Drive the UI. focus-workspace switches Burrow to (and opens) a workspace; focus-tab activates a tab by its pty id, switching to its owning workspace first; new-tab opens a fresh terminal tab in the current (or a given) workspace, optionally running a command. These are UI actions: same file-based request transport as spawn, routed to the origin workspace (ws == BURROW_CWD, always mounted), then pushed to Terminal.vue's poll which dispatches via the shared workspace / terminalTabs stores (wsStore.open, tabsStore.activate/add). new-tab differs from spawn: spawn delegates a sub-agent in the current project; new-tab is a plain tab open that can target any workspace by id. 

burrow focus-workspace 3
burrow focus-tab 42
burrow new-tab --cmd "npm test"
burrow new-tab --ws 3 --cmd htop

burrow worktree <branch> [--base-ref REF] [--path DIR]

Create a git worktree off this workspace's repo on a new or existing branch, and surface it in the Sidebar nested under the parent repo — ready to open and run agents in. Same file-based transport + per-workspace routing as spawn: drops a request dir with kind=worktree plus raw branch/base/path/ws files and a ready marker. Terminal.vue's poll branches on kind: it resolves the parent repo (climbing parent_id if this PTY is itself inside a worktree — no worktree-of-a-worktree), computes the disk path <worktreesDir>/<repo>/<branch> (same convention as the New-worktree dialog), and calls the store's createWorktree. That runs git worktree add in Rust and reloads the workspace list, so the Sidebar updates with no manual refresh.

  • --base-ref REF — base for a new branch (default HEAD; ignored if the branch already exists).
  • --path DIR — override the on-disk location of the worktree.
burrow worktree feat/login                # new branch off HEAD (or check out existing)
burrow worktree hotfix --base-ref main     # new branch based on main
burrow worktree feat/x --path ~/wt/x       # custom on-disk location

burrow wait <token> [--timeout SECONDS] (legacy)

Block until the sub-agent launched with that token finishes its turn, then print its final message to stdout. Exits 1 on timeout. Still available, but prefer collect so you stay productive instead of blocked.

burrow spawn --token t1 claude "summarize what changed in this PR"
burrow wait  t1                 # blocks, then prints the sub-agent's final message
burrow wait  t1 --timeout 600   # give up after 10 minutes

burrow capture <token> (internal)

Not called by hand — it is wired as the spawned Claude's Stop hook (only on tabs that carry a resultToken). It reads the Stop-hook JSON on stdin, pulls transcript_path, extracts the last assistant message (parsed with node, which Claude Code guarantees), then writes <token>.result and <token>.done into $BURROW_SESSION_DIR, and finally calls burrow status done.

Why capture also sends status done: a spawned sub-agent runs with a per-launch --settings whose Stop hook is this capture. Claude Code gives that file precedence and does not also run the globally-installed burrow hook Stop (same event) — so without this the sub-agent's status dot would stick orange after it finished.

burrow status <running|waiting|permission|done|error|session> [meta…]

Reports the current tab's agent state to Burrow's hook HTTP server (a POST /hook with {ptyId,state,…}). Two states carry optional metadata flags: error --detail <error_type> (e.g. billing_error) POSTs {ptyId,state:"error",detail}, and session --model M --source S --title T POSTs {ptyId,state:"session",model,source,title} (source = startup|resume|clear|compact). Metadata values are JSON-escaped. Reads BURROW_PTY_ID + the port. Port resolution is file-first: the live <BURROW_HOME_DIR>/hook.port file is authoritative (the app rewrites it every launch with the current random server port), and BURROW_HOOK_PORT is only a fallback. (Env-first was a bug: a daemon-reattached PTY carries a stale baked-in port and POSTed to a dead server, so the dot never updated.) A no-op outside Burrow. The frontend turns running/waiting/permission/done into the tab's status dot (orange spinner / blue / amber-pulse / green-review).

Sticky states self-heal a dropped POST. If the server is momentarily unreachable (the ~3 s window the app self-reclaims a stale hook.port), a lost status update would leave the dot stuck. So waiting/permission/done/error/session — the states that persist until the next turn — retry up to 3× with a 1 s sleep + hook.port re-read between attempts. running takes a single fast attempt: it fires on every tool call and a lost one self-corrects on the very next event, so it must never block the agent with sleeps (e.g. if the app is fully down).

burrow hook (internal)

The status mechanism — modeled on Superset's single global notify.sh. At startup install_status_hooks merges one persistent hook into each agent's own global config (~/.claude/settings.json, ~/.codex/hooks.json — identical schema), running [ -n "$BURROW_PTY_ID" ] && '…/bin/burrow' hook || true. burrow hook reads the hook JSON on stdin, maps hook_event_name → state and forwards via burrow status:

  • UserPromptSubmit / PostToolUse → running
  • PreToolUse → waiting for the blocking tools AskUserQuestion / ExitPlanMode (a question / a plan to approve), else running
  • PermissionRequest → permission — the agent is blocked on an allow/deny decision (amber dot + Sidebar bell), distinct from a plain waiting
  • Stop → done, except a Stop carrying still-running background_tasks → running. That's an interim stop (the agent parked on background sub-agents and Claude auto-resumes the same session); reporting done there was the premature-completion bug. The background_tasks status check is scoped to that array slice, so an unrelated "status":"running" elsewhere in the JSON can't false-positive.
  • SessionStart → session — forwards model / source / session_title as metadata so Burrow can label the tab (fires on session begin/resume).
  • StopFailure → error — the turn ended on an API error; the error_type field (rate_limit, billing_error, server_error, …) is passed through as detail.
  • Notification → refined by the type field: permission_prompt → permission, idle_prompt → waiting, everything else no-op (the old blanket behavior). Real permission requests still arrive as PermissionRequest; this just rescues clients that surface it as a Notification instead.
  • SubagentStop / SessionEnd → no-op (telemetry, not a turn boundary).

SessionStart / StopFailure and the Notification type discrimination are Claude-only hook events — install_status_hooks registers them in ~/.claude/settings.json only; Codex stays on its existing lifecycle events. The hook server re-emits error/session on the same pty-hook-{id} Tauri event as an object (extra fields passed through); the four legacy states stay bare strings.

Because it's global + env-driven, status works for every agent session — launched-by-button, typed by hand, or reattached after restart — not just per-launch ones. The merge is non-destructive (append + dedupe + .burrow-bak) and a no-op outside Burrow.

The poll never fabricates "busy" for an agent. An agent stays the foreground process whether it's thinking or idle at its own prompt — so equating presence with busy left a permanent orange spinner. Status now comes only from hooks; the 2 s poll only titles agent tabs and clears state when the shell returns (rescuing a Ctrl+C'd agent that never fired done). The poll is sticky per agent session and ignores an empty foreground reading (a daemon race), so child processes the agent spawns and momentary blanks no longer rename the tab to "Terminal N" mid-conversation. XTerm.vue forwards the hook state as one semantic agentState event (running/waiting/permission/done) — a single event has no ordering hazard, so a trailing waiting can't clobber a fresh done.

Dead-PTY watchdog. The one case the shell-return rescue misses is an agent killed or crashed while still foreground — no Stop, no shell return, so an in-flight dot would stick forever. If the foreground reads empty for ≥3 consecutive polls while the last hook state is in-flight (running/waiting/permission), the poll confirms the PTY is actually dead via list_pty_sessions (alive=false) and only then settles the dot. A single empty read is still ignored as a transient race.

How a spawn flows

agent runs:  burrow spawn --token t1 claude "fix tests"
   │  writes  $BURROW_SESSION_DIR/requests/req.XXXXXX/
   │     cmd="claude 'fix tests'"   token="t1"   cwd=<new tab dir>   ws=<workspace dir>
   │     ready                       ← marker written LAST (avoids reading half-written req)
   ▼
Terminal.vue polls take_spawn_requests(cwd) every 1s
   │  Rust claims (deletes) request dirs whose `ws` == this workspace's cwd → routing
   │  → addTab(cmd, { cwd, resultToken })          (Leaf carries cwd + resultToken)
   ▼
new tab runs cmd; status comes from the GLOBAL burrow hook; for a token,
   │     a per-launch --settings adds:  Stop → burrow capture <token>
   ▼
new tab running the sub-agent  ──(on turn end)──►  <token>.result / <token>.done written
   ▲
   └── the spawner keeps working; later `burrow collect t1` reads + consumes the result
       (non-blocking — returns only what has finished). `burrow wait t1` is the legacy
       blocking variant.

Routing to the correct workspace: the request records its workspace dir as ws (from the BURROW_CWD env Burrow sets per PTY). Each Terminal.vue polls with its own cwd, and take_spawn_requests only hands back — and deletes — requests whose ws matches, so the new tab opens in the spawning agent's project. --cwd sets the new tab's own directory via Leaf.cwd.

Two channels (both files)

DirectionMechanismUsed for
CLI → apprequests/req.XXXXXX/ dirs in $BURROW_SESSION_DIRspawn requests (polled by the frontend)
app → CLI<token>.result / <token>.done in $BURROW_SESSION_DIRcollect (or legacy wait) reading capture's output

Request format

Each request is a directory of raw, unencoded files — no escaping, no base64. The command is re-quoted before storage (program name bare, each arg single-quoted) so it re-parses correctly when typed into the new tab: 

requests/req.Ab12Cd/
  cmd     claude 'say hi to the user'      ← re-quoted shell line
  token   t1                               ← may be empty
  cwd     /Users/me/code/app               ← new tab's dir
  ws      /Users/me/code/app               ← workspace routing key
  ready                                    ← empty marker, written last

(Status is a separate channel: globally-installed agent hooks → burrow hook → burrow status → POST /hook on the local HTTP server. No OSC, no tty — see the burrow status / burrow hook sections above. The old OSC 9998 tty protocol is retired.)

tmux shim — CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS

V <app-data>/bin/ (prepended to PATH každého PTY) leží vedle burrow i falešný tmux binary. Překládá tmux API volání na Burrow nativní operace — takže Claude Code's agent teams feature (CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1) funguje v Burrow bez skutečného tmux.

tmux příkazPřeloženo na
tmux new-window [cmd]burrow spawn --tmux-win @N [cmd] — nový tab
tmux split-window [cmd]totéž — Burrow nemá split, otevře nový tab
tmux send-keys -t @N keys EnterPOST /write na hook server → zapíše do PTY daného tabu
tmux new-session / has-sessionno-op (exit 0)
tmux list-* / select-* / set-*minimální fake output nebo no-op
tmux -Vprints "tmux 3.3a"

Mapování window ID (@1, @2…) na Burrow PTY ID drží soubor <app-data>/tmux_wins/@N — zapíše burrow spawn přes flag --tmux-win, přečte send-keys při psaní do tabu.

Proč shim? Agent teams spouští sub-agenty přes tmux new-window. Reálný tmux by otevřel session mimo Burrow — v šedém terminálovém okně, bez status dotů, bez sidebar feedbacku. Shim zachytí volání a přesměruje je do Burrow tabů, kde jsou sub-agenti viditelní a mají plný status tracking.

Caveats

  • The Stop hook fires on every turn end — if a sub-agent pauses to ask a question, that counts as "done". Tell it "don't ask, just do it" in the prompt (or rely on --dangerously-skip-permissions).
  • Result capture is Claude-only (it uses Claude's Stop hook). Other agents spawn fine, but wait only resolves once a .done marker is written.
  • Windows ships the POSIX sh version only for now; a burrow.cmd variant is a future add.
  • Any process in any tab that prints OSC 9999 can open a tab, and any process can POST /hook to the local status server — same trust model (the user's own machine; the server binds 127.0.0.1 only).
