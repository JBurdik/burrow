Burrow — PTY Daemon

Perzistence terminálových sezení přes restarty aplikace

Co to dělá: Claude Code (a jiné agenty) běžící v terminálu přežijí zavření okna Burrow. Druhý den aplikaci znovu otevřeš a terminály jsou tam přesně tam, kde jsi je nechal — živé procesy, žádné replay.

Proč daemon?

PTY (pseudo-terminál) je soubor v OS. Pokud ho vlastní Tauri proces a ten skončí, PTY se zavře a shell/agent s ním. Řešení: přesunout vlastnictví PTY do odděleného procesu (burrow-daemon), který běží nezávisle. Tauri se k němu připojuje přes Unix socket a funguje jako klient.

Stejný přístup používá superset.sh (detached Node.js daemon + node-pty) a tmux (server-klient architektura).

Životní cyklus daemona

1. Tauri setup volá daemon_ensure(data_dir)
   ├── Pokud daemon.sock existuje a daemon.token je čitelný:
   │   └── Zkus ListSessions — pokud ok, daemon běží → připoj se
   └── Jinak: spawn burrow-daemon --env BURROW_DATA_DIR=...
       └── Čekej max 3s na daemon.sock + daemon.token
           └── Připoj se a ulož Arc do DaemonState

2. Při create_pty(id, cwd, cols, rows):
   ├── Pošli CreatePty daemonu
   │   ├── Pokud existing=true: PTY žije z minulé session → reuse
   │   └── Pokud existing=false: daemon spustil nový shell
   └── Spusť start_stream(id): background thread AttachPty → pty-data-{id} Tauri eventy

3. Zavření okna aplikace (onBeforeUnmount v XTerm.vue):
   └── detach_pty(id) → zavře stream thread, NEZABIJE PTY v daemonu
       → Claude Code běží dál v daemonu

4. Explicitní zavření tabu (Terminal.vue closeTab/closePane):
   └── kill_pty(id) → KillPty příkaz → daemon odstraní session

5. Druhý den — restart aplikace:
   ├── daemon_ensure: daemon stále běží → připoj se
   ├── list_pty_sessions: vrátí živé sessions
   ├── Terminal.vue onMounted: načte terminal_tabs z SQLite + daemon sessions
   │   └── Pro živé PTY: makeLeaf(savedId) → create_pty → CreatePty → existing=true → attach
   └── Ring buffer přehrán → uživatel vidí výstup z doby, kdy app byla zavřená

Ring buffer

Každý PTY má v paměti daemona kruhový buffer posledních 512 KB výstupu (VecDeque<Vec<u8>>). Při AttachPty daemon nejprve pošle celý snapshot, pak živá data. Díky tomu uživatel po reconnectu vidí vše, co se stalo v době, kdy byl odpojený.

SQLite změny

Do tabulky terminal_tabs přibyla dvě pole (idempotentní migration v init_db):

ALTER TABLE terminal_tabs ADD COLUMN pty_id INTEGER;  -- id PTY v daemonu
ALTER TABLE terminal_tabs ADD COLUMN cwd TEXT;         -- poslední pracovní adresář

pty_id se ukládá při každém persist() v Terminal.vue. Při obnově: pokud pty_id je v daemon ListSessions jako alive=true, leaf dostane ten samý id → create_pty ho připojí k existující session.

Kde najít daemon binary v development

V development módu Tauri neví o burrow-daemon. Daemon se nenajde automaticky — nastav env proměnnou: 

# Nejprve build daemona:
cargo build --bin burrow-daemon

# Pak spusť Tauri s path k binarce:
BURROW_DAEMON_BIN=target/debug/burrow-daemon pnpm tauri:dev

find_daemon_binary v lib.rs zkouší (v pořadí): env BURROW_DAEMON_BIN, pak cestu vedle aktuálního exe, pak Tauri resource dir (pro production bundle).

Production bundle

Před buildem produkční verze je potřeba zkopírovat burrow-daemon do stejné složky jako hlavní binary. Tauri bundler pak oba soubory zabalí do Burrow.app/Contents/MacOS/. 

# Build obou binarek:
cargo build --release --bin agentic-ide
cargo build --release --bin burrow-daemon

# Production bundle (pnpm tauri:build to udělá automaticky přes beforeBuildCommand):
pnpm tauri:build

Bezpečnost

Daemon přijímá pouze lokální Unix socket spojení (žádná síť). Auth token je 32 náhodných bajtů z /dev/urandom, uložený v daemon.token (čitelný jen aktuálním uživatelem díky chmod). Všechna spojení musí začít Auth příkazem.
