Mission Control POC

A tank-style task dashboard inside Burrow — an in-window view (not a separate window) that drives headless claude PTYs as disposable tasks. Same model as tank, but on Burrow's native Rust PTY instead of tmux, with tasks persisted in the shared workspaces.db.
Status: proof of concept. Scope: spawn + list + live terminal + result capture + sequential queue + follow-up conversation + SQLite persistence + live PTY reconciliation + image attachments + send-to-tab + per-task git worktree. Remaining backlog (AI titles, live streaming, schedules, PR button, notifications) lives in docs/plans/mission-control.todo.md. Read the tank analysis first — this doc assumes you know tank's three-channel model and only describes how Burrow realises it.

How to use it

  1. Click the 🚀 button in the ActivityBar (left rail, below the git icon) — it toggles ui.mode between 'mission' and 'terminal', swapping the center pane in place (same mechanism as the Git panel). No new window.
  2. Scope = the active workspace, chosen in Burrow's sidebar (no picker of its own). The rail mirrors it read-only (icon + name + cwd). With no active workspace the whole view is a centered blank state (Pick a workspace) — nothing to scope a task to, so ＋ New task can't fire.
  3. Hit ＋ New task → a composer modal: prompt + model. Working dir is implied by the active workspace (shown as a cwd line, not an editable field) → Run now (or ⌘↵), or Add to queue. Switching workspace in the sidebar re-targets the composer at the new cwd (watch(activeWs)).
  4. The new task is selected automatically; its conversation shows in the center panel and is grouped under its workspace in the left rail.
  5. Type in the send & continue bar at the bottom to follow up; stop interrupts the current turn (Ctrl+C).
  6. Terminal attaches a live xterm; Delete kills + removes the task. (Chrome uses Phosphor icons throughout, the same set as the rest of Burrow — no emoji glyphs.)
  7. Tasks survive restarts (stored in SQLite). A restored task is read-only — its PTY died with the previous app session, so live attach & follow-up are gone, but the conversation persists.

Spawn flow

Run prompt │ ├─ id = crypto.randomUUID() ← also Claude --session-id ├─ ptyId = PTY_BASE(2_000_000) + counter ← own id-space, never collides │ with the main window's PTYs ├─ wireTask(task) ← attach pty-data + pty-hook listeners ├─ create_pty(ptyId, cwd, 120, 34) ← headless shell (Rust daemon) └─ setTimeout(450ms): ← wait out the shell rc write_pty(ptyId, "claude --session-id <id> --model X '<prompt>'\n")
Why PTY_BASE = 2_000_000. PTY ids come from a per-window counter (src/lib/ptyId.ts) starting at 0, and the Rust daemon keys its PTY map by a single global id. The mission window is a separate webview = separate JS context = its own counter from 0 → it would collide with the main window's tabs. Offsetting into a high range keeps the id-spaces disjoint.
Why the 450 ms delay. create_pty spawns the user's login shell; writing the claude line before the rc finishes can get the line swallowed. A short delay is the POC's pragmatic fix (tank sidesteps this entirely by exec-ing claude as the tmux command).

Status — reusing Burrow's hook server

This is the cleanest win over tank: no new hook wiring. create_pty already exports BURROW_PTY_ID=ptyId into the PTY env, and Burrow globally installs a status hook into ~/.claude/settings.json. So a mission-spawned claude reports status through the exact same path as a hand-launched Burrow tab:

claude lifecycle event → global hook: burrow hook (maps event → running|waiting|done) → burrow status <state> (POST to the tiny_http hook server) → start_hook_server re-emits → Tauri event pty-hook-{ptyId} → MissionControl.vue listener → task.status = state

On done the listener schedules captureResult() ~600 ms later (transcript flushes a beat after Stop) and calls pumpQueue() to start the next queued prompt.

Follow-up — continuing a task conversation

Each task is a conversation, not a one-shot. task.turns[] holds alternating user / assistant entries; the card renders them and shows a Follow up… input at the bottom while the PTY is alive.

Simpler than tank. tank kills the tmux session on Stop, so a follow-up needs claude --resume <id> (a fresh process). Mission Control never kills the PTY on done — claude is still sitting interactively at its input prompt. So a follow-up is just write_pty(ptyId, text + "\r"). The global UserPromptSubmit hook flips the status dot back to running automatically; the next Stop appends the new assistant turn. 
function sendFollowup(t) {
  t.turns.push({ role: 'user', text });
  t.status = 'running';
  const line = text.replace(/\r?\n/g, ' ') + '\r';   // REPL submits on Enter
  write_pty(t.ptyId, bytes(line));
}
Newline caveat: claude's REPL submits on Enter, so a raw \n in a follow-up would split the message into several submits. The POC collapses newlines to spaces. A faithful multi-line input would send each newline as the REPL's "insert newline" key (Shift+Enter equivalent) instead.

Live terminal — attach on demand

tank gets free scrollback from tmux attach. A headless native PTY has no pane to re-read, so Mission Control keeps its own replay buffer:

  • From spawn, the pty-data-{ptyId} listener appends decoded output into buffers[ptyId] (capped at 256 KB).
  • Opening the modal creates a fresh xterm.js + FitAddon, writes the buffer, then streams live bytes.
  • Keystrokes → write_pty; resize → resize_pty; a ^C button sends 0x03.

Queue — sequential runner

function activeCount() {            // running | waiting tasks that are alive
  return tasks.filter(t => t.alive && (t.status==='running'||t.status==='waiting')).length;
}
function pumpQueue() {              // fill up to maxConcurrent slots
  while (queue.length && activeCount() < max(1, maxConcurrent))
    spawnTask(queue.shift());
}

pumpQueue() is called on enqueue, on a task's done, and on manual stop. Default maxConcurrent = 1 (true sequential); bump it for parallel fan-out.

Persistence — shared workspaces.db

Tasks live in the same SQLite DB as workspaces and terminal tabs — the mission_tasks table (id, workspace_id FK, pty_id, title, cwd, model, status, turns as JSON, created_at). The frontend upserts a row on every state change (saveTask) and loads all rows on mount (loadTasks).

// per-task save — snake_case keys match the serde struct
upsert_mission_task({ task: { id, workspace_id, pty_id, title, cwd,
                              model, status, turns: JSON.stringify(turns),
                              created_at } })
  • Workspace-scoped: each task records the workspace whose path === cwd (else the active workspace). The rail groups tasks under their workspace name — Burrow workspaces are tank's "projects".
  • Restored = read-only: live PTYs aren't serialisable, so a task loaded from DB is alive: false — conversation persists, but live-attach & follow-up are gone (the daemon PTY died with the previous app session).
  • PTY id continuity: on load, ptyCounter is bumped past the max restored pty_id so new tasks never collide in the daemon's global PTY map.

Known POC limits

LimitWhy / fix later
450 ms write delayRace against shell rc. Proper fix: have the daemon exec claude directly as the PTY command (like tank).
Single-quote prompt escapingshquote() handles ' only; fine for POC. Long prompts could move to a temp file like tank.
No live PTY reconciliationRestored tasks are read-only even if the daemon still holds their PTY. A future pass could rebind via list_pty_sessions (tank does this with reconcile_previews).
No image attachments / AI titles / PR buttonsIndependent surface, not wired yet. See tank's feature list for the recipes.
