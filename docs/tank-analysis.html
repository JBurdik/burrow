Why this doc exists. tank is the design tank's Mission Control POC was modelled on. This page captures how tank works so any agent can re-derive or extend the pattern later without re-reading tank's 5k-line api.py. For the Burrow port, see Mission Control →.

What tank is

A single-box web app that runs Claude Code as a fleet of disposable tasks. You type a prompt in a browser dashboard; tank spins up a fresh claude process in its own sandbox, streams its status back to the GUI, lets you attach a live terminal, and captures the final answer when the turn ends — then disposes the session.

  • Backend: one file, api.py — FastAPI + SQLite (rusqlite-free, stdlib sqlite3) + httpx.
  • Frontend: one file, static/index.html — ~5k lines, no framework, no build step. xterm.js from a CDN.
  • Process model: every task is a tmux session running claude --session-id <uuid>.
  • Deploy: systemd service (tank.service) as a dedicated unprivileged user; install.sh / deploy.sh.

The three channels

tank never invents an AI protocol. Claude is a plain CLI in tmux; three independent channels move data between it and the browser:

ChannelDirectionCarries
tmux + PTYapp → claudeThe prompt (typed in as claude's CLI arg) + live keystrokes on attach.
hook.sh → RESTclaude → appLifecycle events (start / stop / etc.) via Claude Code hooks.
SSE / WebSocketapp → browserSSE = state patches; WebSocket = live terminal bytes.

1 · Spawning a task prompt → claude

Key functions: _spawn_claude(), spawn_first_turn(), tmux_new_session().

browser POST /tasks { prompt, project, model } │ ├─ task_id = uuid4() ← also becomes claude --session-id ├─ write prompt to /tmp/tank-prompts/<id> ← tmux command string caps at ~16KB ├─ build settings.json with per-task hooks └─ tmux new-session -d -s <name> -c <cwd> \ 'P=$(cat promptfile); rm promptfile; exec claude --session-id <id> "$P"' ← exec → the pane *is* claude
  • Prompt rides in a file, not the tmux command (tmux truncates long new-session strings). The shell reads it into claude's argv (bounded by ARG_MAX ~2MB) then deletes it.
  • exec claude replaces the shell so the tmux pane is the claude process — a clean pkill / attach target.
  • Continuations reuse the session: claude --resume <uuid>.
  • Model is fixed at spawn (--model); a resumed session pins to the model it was saved with.

2 · Status & results claude → app

tank installs a hook bridge at user level in ~/.claude/settings.json. Claude calls it on every lifecycle event.

hook.sh

KIND="${1:-unknown}"                 # event kind from argv (Stop, SessionStart…)
PAYLOAD="$(cat)"                     # full event JSON on stdin
TASK_ID=$(echo "$PAYLOAD" | jq -r '.session_id')   # == the --session-id we set
BODY=$(jq -nc --arg k "$KIND" --argjson p "$PAYLOAD" '{kind:$k, payload:$p}')
curl -sfm 5 -X POST "http://127.0.0.1:${PORT}/tasks/${TASK_ID}/events" \
     --data-binary "$BODY" || true   # never block claude on hook delivery
The crux: session_id in the payload equals the task_id tank passed via claude --session-id <task-id>. So the hook knows which task it belongs to with zero extra wiring. Hooks for sessions tank didn't spawn POST to an unknown id and silently fail at the FK layer — harmless.

Server side — handle_event()

  • SessionStart → DB status = running.
  • Stop → read the JSONL transcript, extract the last assistant message (read_last_assistant_text()), save it as the task result, then kill the tmux session.
  • Every event calls broadcast() → pushes to per-task SSE queues.
  • read_last_api_error() detects synthetic API-error messages (claude tags isApiErrorMessage) so a 429/529 surfaces as an error, not a result.

The transcript

Claude writes every session to ~/.claude/projects/<slug>/<session_id>.jsonl, where <slug> is the cwd with / and . replaced by -. Each line is a JSON record; assistant text lives in message.content[].text for type=="assistant" blocks. tank reads it bottom-up for the last one.

3 · The GUI app → browser

SSE — state

// global re-list when anything happens anywhere
const globalEvents = new EventSource('/stream');
// per-task detail (status, result) patched in-place
detailEvents = new EventSource(`/tasks/${id}/stream`);

Server holds a set of asyncio.Queues per task (sse_subscribers); broadcast() fans an event to all of them; EventSourceResponse drains the queue to the browser. The dashboard re-renders dynamic regions in place rather than reloading.

WebSocket — live terminal

Endpoint @app.websocket("/tasks/{task_id}/terminal/{turn_num}") → terminal_ws(). It bridges an xterm.js client to the tmux pane via a forked PTY:

browser xterm.js ⇄ WS /tasks/<id>/terminal/<turn> ⇄ pty.fork() → tmux attach -t <sess>
  1. pty.fork(); child does os.execvpe("tmux", ["tmux","attach","-t",sess], env) with TERM=xterm-256color (systemd has no TERM).
  2. Parent: loop.add_reader(fd, …) → os.read(fd) → ws.send_text() → xterm renders.
  3. Keystrokes: ws.receive() → os.write(fd, data) → tmux → claude.
  4. Resize: xterm sends a custom frame \x1b__RESIZE__{cols,rows} → fcntl.ioctl(fd, TIOCSWINSZ, …).
tmux attach gives the full pane buffer for free, so re-attaching shows existing scrollback. A native-PTY clone (Burrow) must replay its own buffer — see the port doc.

Clone recipe — minimum viable tank

To rebuild the core on any stack, you need exactly these five moving parts:

  1. An id you control end-to-end. Generate session_id = uuid; pass it as claude --session-id. It re-appears in every hook payload and names the transcript file — your single join key.
  2. A PTY host for claude. tmux, or a native PTY. Type claude --session-id <id> [--model X] '<prompt>' into it. Long prompts → file, not the command line.
  3. A lifecycle hook. Register a hook in ~/.claude/settings.json that ships {kind, payload} to your backend (HTTP, file, socket — anything). Map SessionStart→running, Stop→done.
  4. A result reader. On Stop, read ~/.claude/projects/<slug>/<id>.jsonl bottom-up for the last assistant text block.
  5. Two browser channels. Push channel for state (SSE/events). Bidirectional channel for the live terminal (WebSocket ↔ PTY master fd ↔ xterm.js, with a resize frame).
Everything else — queue, cron, previews, AI titles, PR buttons — is independent product surface layered on top. The five parts above are the irreducible engine.

What to keep vs change when porting

tankVerdict for a native-PTY host
tmux session per taskReplace — native PTY is faster, no tmux dep. But lose free scrollback-on-reattach → replay a buffer yourself.
hook.sh → curl → RESTKeep the idea; the channel can be whatever the host already has (Burrow: hook server → pty-hook-{id} event).
Transcript readingKeep verbatim — it's a Claude Code contract, host-independent.
SSE for stateKeep, or use the host's reactive store (Burrow: Vue/Pinia, no SSE needed in-process).
pty.fork + tmux attach for terminalReplace with the host's existing PTY data stream + xterm replay.
5k-line single HTMLReplace with components if the host has a framework.
